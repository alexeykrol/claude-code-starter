# Claude Code Starter v6.1.0

## Summary

`v6.1.0` is the first published release of the v6 series. It introduces two major capabilities:

1. **Content-aware framework** — projects whose product is text (books, courses, knowledge bases, documents, transcripts) get a dedicated layer of rules, skills, and agents.
2. **Optional global layer** — framework can be installed once into `~/.claude/` and reused across every project, with a `/setup-project` skill that bootstraps a project from inside Claude Code (no manual installer download).

The default install requires no flags: `bash init-project.sh` auto-detects project type, content type, and scenario.

This is the first GitHub Release of the v6 series; v6.0.0 was an internal milestone within development. All v6.0 changes are included here.

## What's New

### Content-aware framework

- `templates/content/` with content-specific `CLAUDE.md` and `SNAPSHOT.md`
- 5 rules: `content-pipeline`, `content-quality`, `source-management`, `content-formats`, `content-commit-policy`
- 7 skills: `research`, `outline`, `write-content`, `review-content`, `enrich`, `content-index`, content-aware `housekeeping`
- 3 agents: `writer`, `editor`, content `reviewer` (28-point review checklist)
- 5 content unit templates: `chapter`, `lesson`, `transcript`, `article`, `document`
- Starter directory layouts for `book`, `course`, `knowledge-base`, `documents`, `transcripts`

### Auto-detection

- Project type: `code` / `content` / `hybrid` / `unknown` (file extensions, directory signals, manifest detection)
- Content type: `book` / `course` / `knowledge-base` / `documents` / `transcripts` / `mixed` (directory layout)

### Safe additive `CLAUDE.md` merge

- New Python helper `scripts/lib/merge_claude_md.py` (stdlib only)
- Section-aware merge with similarity scoring
- Hard conflicts halt the install and produce a concrete resolution proposal in `.claude/CLAUDE.md.merge-proposal.md`
- 26 unit tests
- Verified on 16 real projects: every custom user section preserved across migration

### Backup & rollback

- Each install/migrate snapshots `CLAUDE.md`, `settings.json`, `SNAPSHOT.md`, `manifest.md`, `.gitignore` to `.claude/backup-TIMESTAMP/`
- `bash init-project.sh --rollback` restores the latest backup
- `bash init-project.sh --apply-proposal` applies a CLAUDE.md merge proposal after manual review

### Hybrid mode

- Both code and content layers coexist
- `code-reviewer` and `content-reviewer` kept separate
- `content-housekeeping` replaces `code-housekeeping` (universal)

### Global layer (new in 6.1.0)

- `scripts/install-global.sh` adds framework rules / skills / agents to `~/.claude/` additively
- Backups `~/.claude/` to `~/.claude/.backup-TIMESTAMP/` before any change
- Preserves existing user `~/.claude/CLAUDE.md` content via the same additive merger
- Records framework checkout location in `~/.claude/framework-source-path`
- New skill `/setup-project` (in `templates/global/skills/`) bootstraps a project from inside Claude Code without downloading anything
- `--rollback` and `--dry-run` supported

### New CLI surface

- `--type code|content|hybrid|auto`
- `--content-type book|course|knowledge-base|documents|transcripts|mixed`
- `--rollback`
- `--apply-proposal`
- `--force`

### Manifest extension

- `project_type` and `content_type` fields in `manifest.md`

## Operational Changes

### Public installer

- `init-project.sh` (root) bumped to v6.1.0
- Pass-through for new flags
- Special modes (`--rollback`, `--apply-proposal`) bypass scenario detection and route directly to `migrate.sh`

### Internal scripts

- Shared install logic extracted to `scripts/lib/install_common.sh` (single source of truth for install + migrate)
- `scripts/init-project.sh` rewritten as thin parser + dispatcher
- `scripts/migrate.sh` rewritten as additive migrator with auto-detect, backup, conflict-aware merge

### Build

- `scripts/build-release.sh` now bundles `templates/content/`, `templates/global/`, and `scripts/lib/` into `framework.tar.gz`
- Strips `__pycache__` and `.DS_Store` from staging dir before tar

## Upgrade Notes

### From v5.x

The installer detects existing v5 installs as `upgrade` scenario and runs additive migration. Existing custom `CLAUDE.md` content is preserved through the new merger.

If you have a custom `CLAUDE.md`, the merge will be additive: framework sections are added, your sections are kept. Hard conflicts (rare) are written to `.claude/CLAUDE.md.merge-proposal.md` for manual resolution.

### Choosing install path

- **Quick / one-off:** download `init-project.sh`, run it (Variant 1 in README).
- **Local checkout:** clone the repo, run `init-project.sh` from the checkout (Variant 2).
- **Reuse across all projects:** install global layer, then use `/setup-project` skill in any new folder (Variant 3, new in 6.1.0).

## Known Limitations

- `--apply-proposal` falls back to a manual instruction if the Python merger does not yet implement the apply-proposal subcommand (placeholder for future iteration)
- Auto-detection uses `find -maxdepth 4`; deeply nested content may be undercounted
- Hybrid mode picks the content `CLAUDE.md` template (broader); pure-code projects with hybrid signals may want to override via `--type code`
- `migrate.sh` still depends on `python3` for safe JSON merge of `.claude/settings.json`
- GitHub Release publishing is still a manual step

## Compatibility

- Backward compatible with v5.x installer flags (`--name`, `--mode`, `--template`)
- Existing v5 projects can migrate by running the v6.1.0 installer; no data loss expected (verified on 16 real projects)
